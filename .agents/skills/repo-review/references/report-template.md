# レポート出力テンプレート

レビュー結果は以下の構造に厳密に従って Markdown で出力する。セクション順・見出しレベルを守ること。該当する指摘が無い重要度の小見出しは省略してよい（例: Critical が無ければその小見出しごと省く）。各指摘は `file:line` かコマンド出力を根拠に持つこと。

重要度: 🔴 Critical / 🟠 Warning / 🟡 Info / 🟢 Good

---

```markdown
# リポジトリレビュー — <リポジトリ名 / 対象>

## サマリー

- **対象**: <リポジトリ名・主な技術スタック（Next.js x.y / React / TypeScript など）>
- **パッケージマネージャ**: <pnpm / npm / yarn / bun>
- **レビュー観点**: 開発者目線 / 運用者目線

### 検証コマンド結果

| コマンド            | 結果                | 備考                 |
| ------------------- | ------------------- | -------------------- |
| `<pm> lint`         | ✅ / ❌ / ⏭️ skip   | <要点・失敗理由など> |
| `<pm> format:check` | ✅ / ❌ / ⏭️ skip   |                      |
| `<pm> build`        | ✅ / ❌ / ⏭️ skip   |                      |
| `<pm> test`         | ✅ / ❌ / ⏭️ 未設定 |                      |

### 指摘サマリー

<全体の総評を2〜4文で。最も優先すべきテーマと、このリポジトリの性格（テンプレ/本番アプリ等）を踏まえた一言>

---
```

## 運用ノート

- **件数の整合**: 指摘サマリーの件数と本文の指摘数を一致させる。
- **根拠主義**: 「該当」には必ず具体的な `file:line`、または「`<pm> build` の出力」のような検証の出どころを書く。「リポジトリ全体」の指摘（例: テスト未導入）は該当にその旨を明記する。
- **重要度の正しさ**: 好みの問題を Critical にしない。本番リスク（漏洩・障害・データ損失）だけが Critical。
- **良い点を必ず1つ以上**: 各レンズで 🟢 を最低1つ挙げ、是正一辺倒のレポートにしない。
