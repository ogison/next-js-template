# 運用者目線チェックリスト

本番で動かし続ける立場で「安全に出せるか・回し続けられるか・壊れたとき気づけるか」を見る観点集。各項目は網羅の手がかり。指摘には必ず該当ファイル（`file:line`）かコマンド出力を根拠に添える。

## 1. CI/CD

- `.github/workflows/` 等に CI があるか。lint・format・build・test のどこまでをカバーしているか。**テストが CI に無い**のはよくある欠落。
- 依存キャッシュ（`actions/setup-node` の `cache` 等）で実行時間を抑えているか。
- **最小権限の原則**: workflow の `permissions` が必要最小限に絞られているか。`contents: read` 等の明示が無く既定の広い権限のままだと Warning。
- トリガー条件（PR / push / branch）が適切か。
- ブランチ保護（必須チェック・レビュー必須）が前提とされているか。設定はリポジトリ外だが、CI がそれを意図しているか読み取る。
- ロックファイルの整合性検証（`--frozen-lockfile` / `npm ci`）で再現性を担保しているか。

## 2. 依存更新の自動化

- Dependabot（`.github/dependabot.yml`）または Renovate の設定があるか。スケジュール・対象エコシステム・PR 上限。
- メジャー更新の扱い（自動マージ対象外にして手動レビューにしているか）。
- **auto-merge の安全性**: 自動マージ workflow がある場合、`pull_request_target` を使っていないか要確認。`pull_request_target` は fork PR でもベースリポジトリの権限・シークレットで実行されるため、チェックアウトやビルドの仕方次第で**権限昇格・シークレット漏洩のリスク**がある（Dependabot 限定・コードを実行しない構成なら許容範囲だが、根拠を明記）。
- 自動マージの条件（semver レベル・ラベル・CI 成功必須）が安全側か。

## 3. セキュリティ

- シークレットスキャン（GitHub secret scanning / gitleaks 等）や SAST（CodeQL）が有効か。無ければ Info〜Warning。
- 依存の脆弱性チェック（`npm audit` / `pnpm audit`、Dependabot alerts）が回っているか。実際に audit を走らせて結果を確認してもよい。
- コミット済みのシークレット・APIキー・`.env` ファイルが無いか。`.gitignore` で `.env*` が除外されているか。
- workflow での `secrets` の扱い、サードパーティ Action のバージョン固定（タグではなく SHA ピン留めが望ましい）。
- 依存の `postinstall` スクリプト等、サプライチェーン上のリスク。

## 4. 環境変数 / シークレット

- `.env.example`（またはそれに類する雛形）があり、必要な環境変数が文書化されているか。**不在はよくある欠落**で、オンボーディングと本番設定ミスの温床。
- 環境変数の**実行時検証**（zod / `@t3-oss/env-nextjs` 等のスキーマ）があるか。無いと未設定や型違いが本番で初めて顕在化する。
- クライアントに露出してよい変数（`NEXT_PUBLIC_` 接頭辞）とサーバ専用変数の区別が正しいか。秘密情報が `NEXT_PUBLIC_` に乗っていないか（**漏洩リスク = Critical**）。

## 5. デプロイ

- デプロイ先の前提（Vercel / Docker / セルフホスト）と、それに応じた設定の有無。
- Vercel 前提なら `next.config` の自動検出に任せているか、`vercel.json` で明示しているか。
- コンテナ運用なら `Dockerfile` の有無・マルチステージビルド・`output: "standalone"` の利用・`.dockerignore`。
- ビルド成果物・出力モードの設定が運用形態と一致しているか。
- ヘルスチェック用エンドポイントの有無。

## 6. 可観測性 / 監視

- 構造化ロギングの方針があるか（`console.log` 散在で済ませていないか）。
- エラートラッキング（Sentry 等）・APM・アナリティクスの導入有無。
- 本番でのデバッグ手段（リクエスト ID、ログレベル制御）が考えられているか。
- 小規模/テンプレ段階では未導入が普通なので、重大度は文脈に応じて（多くは Info）。

## 7. ランタイム整合性

- Node バージョンの指定が一箇所に集約され矛盾していないか（`.node-version` / `.nvmrc` / `package.json` の `engines`）。CI が使う版とローカル指定がずれていないか。
- パッケージマネージャの固定（`packageManager` フィールド / `corepack`）。CI とローカルで版が揃うか。
- lockfile がコミットされ、複数のパッケージマネージャの lockfile が混在していないか。

## 8. リリース運用 / その他

- バージョニング・CHANGELOG・リリースタグの運用があるか（ライブラリ/長期運用なら重要、テンプレなら Info）。
- CODEOWNERS によるレビュー担当の自動割り当て。**不在はよくある欠落**。
- Issue / PR テンプレート、コントリビューションガイド。
- ライセンス表記の有無と妥当性。
